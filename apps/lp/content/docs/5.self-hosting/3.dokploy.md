---
title: Deploy with Dokploy
description: Run Shelve with Docker Compose, Dokploy routing, and a private PostgreSQL network
---

Shelve can run on Dokploy with the root `docker-compose.yml`. The app container joins both the Dokploy routing network and a private database network; PostgreSQL only joins the private network.

## What the Compose file does

- Builds Shelve with placeholder build-time values and `SKIP_ENV_VALIDATION=true`.
- Reads real secrets at container runtime from Dokploy's Env tab.
- Creates an internal PostgreSQL service and persistent Docker volume.
- Waits for PostgreSQL, runs pending Drizzle migrations, then starts the Nuxt/Nitro server.
- Exposes port `3000` to Dokploy without publishing PostgreSQL publicly.

## Dokploy setup

1. Create a Docker Compose application in Dokploy from this repository.
2. Use the repository root as the Compose directory.
3. Keep the default Compose file path: `docker-compose.yml`.
4. Add the required variables in the Dokploy Env tab.
5. Point your Dokploy domain to the `shelve` service on port `3000`.

Dokploy writes Env tab values to a `.env` file next to the Compose file. The Compose file references those values explicitly with `${VAR_NAME}` and passes them to the container.

## Required variables

```bash
# PostgreSQL, used by the internal postgres service
POSTGRES_PASSWORD=generate-with-openssl-rand-hex-32

# Shelve runtime secrets. Never change these after first production use.
NUXT_SESSION_PASSWORD=generate-with-openssl-rand-base64-48
NUXT_PRIVATE_ENCRYPTION_KEY=generate-with-openssl-rand-base64-48
```

Generate suitable values:

```bash
openssl rand -hex 32
openssl rand -base64 48
openssl rand -base64 48
```

You do not need to set `DATABASE_URL` when using the bundled PostgreSQL service. The entrypoint builds it from `POSTGRES_HOST`, `POSTGRES_PORT`, `POSTGRES_DB`, `POSTGRES_USER`, and `POSTGRES_PASSWORD`, URL-encoding the password safely.

## Secret handling

The Docker build does not require production secrets. The image is built with `SKIP_ENV_VALIDATION=true`, and NuxtHub build-time database migrations are disabled so the build never needs `DATABASE_URL`.

Runtime secrets are still required by the running container. When you set secrets in Dokploy's Env tab, Docker exposes those values to the container as environment variables. They are not baked into the image, but a Docker host administrator can inspect them on the running container.

For stricter secret handling, mount secret files and set the matching `*_FILE` variable instead of the plain value:

```bash
POSTGRES_PASSWORD_FILE=/run/secrets/shelve_postgres_password
NUXT_SESSION_PASSWORD_FILE=/run/secrets/shelve_session_password
NUXT_PRIVATE_ENCRYPTION_KEY_FILE=/run/secrets/shelve_encryption_key
NUXT_OAUTH_GITHUB_CLIENT_SECRET_FILE=/run/secrets/shelve_github_oauth_secret
NUXT_OAUTH_GOOGLE_CLIENT_SECRET_FILE=/run/secrets/shelve_google_oauth_secret
NUXT_PRIVATE_GITHUB_PRIVATE_KEY_FILE=/run/secrets/shelve_github_private_key
NUXT_PRIVATE_RESEND_API_KEY_FILE=/run/secrets/shelve_resend_api_key
NUXT_PRIVATE_RESEND_WEBHOOK_SECRET_FILE=/run/secrets/shelve_resend_webhook_secret
```

Do not set both `NAME` and `NAME_FILE` for the same secret.

## Authentication

Configure at least one login method.

Email OTP:

```bash
NUXT_PRIVATE_RESEND_API_KEY=re_your_resend_key
NUXT_PRIVATE_SENDER_EMAIL=noreply@example.com
```

GitHub OAuth:

```bash
NUXT_OAUTH_GITHUB_CLIENT_ID=your_github_client_id
NUXT_OAUTH_GITHUB_CLIENT_SECRET=your_github_client_secret
```

Google OAuth:

```bash
NUXT_OAUTH_GOOGLE_CLIENT_ID=your_google_client_id
NUXT_OAUTH_GOOGLE_CLIENT_SECRET=your_google_client_secret
```

## Recommended production variables

```bash
NUXT_PRIVATE_ALLOWED_ORIGINS=https://shelve.example.com
NUXT_PRIVATE_ADMIN_EMAILS=admin@example.com
```

For GitHub App sync, also set:

```bash
NUXT_PUBLIC_GITHUB_APP_NAME=your-github-app-name
NUXT_PRIVATE_GITHUB_PRIVATE_KEY=base64_pkcs8_private_key
```

## Networks

The app joins Dokploy's external `dokploy-network` and Shelve's private `shelve-db` network. The `shelve-db` network is marked `internal: true`, so the database is reachable by Shelve but not exposed through Dokploy routing.

## Migrations

Automatic migrations are enabled by default:

```bash
SHELVE_AUTO_MIGRATE=true
```

Disable them only if you run migrations separately:

```bash
SHELVE_AUTO_MIGRATE=false
```

## External database

To use an external PostgreSQL database instead of the bundled service, set `DATABASE_URL` in the Env tab and remove or ignore the `postgres` service. If your database password contains reserved URL characters, URL-encode it in `DATABASE_URL`.
